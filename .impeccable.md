# Impeccable Design Context

## Design Context

### Users

Asterel docs serve three overlapping reader groups:

- operators who want to run a Discord-first companion runtime locally;
- contributors who need architecture, security, and release-boundary context;
- research or citation readers who need claims, evidence, limitations, and
  reproducibility without private development history.

Readers are usually in a focused documentation context, not a marketing browsing
context. They need trust, traceability, and fast comprehension more than spectacle.

### Brand Personality

Quiet, evidence-first, and warm. The interface should feel like a careful field
notebook for a technical companion runtime: calm, legible, restrained, and
slightly tactile. It should not feel like a generic AI SaaS landing page.

### Aesthetic Direction

Warm paper editorial documentation with flat edges, ink-like contrast, sparse
accent color, and measured typography. Light and dark themes both matter because
readers may use the docs during setup, debugging, or late-night research reading.

Avoid neon AI palettes, glassmorphism, gradient text, hero-stat blocks, and
repeated icon-card grids. Use structure, linework, and precise copy instead of
decorative effects.

### Design Principles

1. Evidence before persuasion: every page should make claims and verification
   boundaries easy to inspect.
2. Quiet hierarchy: typography, spacing, and linework should guide attention
   without shouting.
3. Public-safe by default: docs should not depend on private notes, private logs,
   or local-only agent context.
4. Dense content must remain readable: tables, commands, and research results need
   mobile fallbacks or summaries.
5. Distinct without AI slop: retain the warm notebook identity while avoiding
   template card grids, glow effects, and generic SaaS polish.
